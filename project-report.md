# 浙江工商大学研究生奖学金评选系统项目报告

## 一、项目概述

本项目为“浙江工商大学研究生奖学金评选系统”，面向信息与电子工程学院研究生综合测评与学业奖学金评选业务场景开发。系统依据《浙江工商大学奖学金实施办法》及学院研究生综合测评细则，实现学生综测填报、辅导员审核、管理员奖学金项目配置、系统自动排名、公示结果查询等功能。

系统采用前后端分离架构：后端基于 Java 17、Spring Boot 3.2 和 MyBatis-Plus 构建 RESTful API；前端基于 React 18、Vite 和 Ant Design 实现多角色业务界面；数据库使用 H2 文件模式，支持零配置启动、自动建表和演示数据初始化，便于教学展示与本地部署。

## 二、项目背景与意义

研究生奖学金评选通常涉及学生材料填报、辅导员审核、成绩统计、综合测评计算、奖学金等级分配、公示等多个环节。传统人工或表格化处理方式存在以下问题：

1. 数据分散，学生材料、课程成绩、审核意见和最终奖项之间缺少统一管理。
2. 评分规则复杂，德育、智育、素质拓展、创新模块权重和计分口径容易出现人工误差。
3. 排名与奖学金等级分配需要按比例计算，人工处理效率低且可追溯性不足。
4. 公示与查询依赖人工整理，信息更新不及时。

本系统通过信息化方式将评选流程固化为可操作、可审核、可追溯的线上流程，提高奖学金评选工作的规范性、公平性和效率。

## 三、需求分析

### 3.1 用户角色

系统包含三类核心角色：

| 角色 | 主要职责 |
|------|----------|
| 学生 | 查看个人信息，填报综合测评材料，上传附件，提交综测，查看推荐等级，申报奖学金 |
| 辅导员 | 查看所带学生信息，审核学生综测材料，审核奖学金申请，支持批量通过 |
| 管理员 | 管理学年，创建奖学金项目，配置等级与规则，执行排名，发布公示，导入学生和成绩数据 |

### 3.2 功能需求

系统主要功能包括：

1. 用户登录与身份认证：支持学生、辅导员、管理员登录，使用 JWT 维护会话状态，支持修改密码。
2. 综合测评填报：学生可填报德育、素质拓展、创新模块材料，系统根据规则自动计算单项分数。
3. 智育成绩计算：系统根据课程成绩与学分计算加权平均分，作为智育模块成绩。
4. 材料审核：辅导员可审核学生提交的材料，审核通过或驳回后影响综测结果计算。
5. 奖学金项目管理：管理员可创建奖学金项目，配置奖项等级、金额、比例和申请条件。
6. 排名与等级分配：系统根据综合测评分数自动排名，并按配置比例分配奖学金等级。
7. 奖学金申报与审核：学生可申报符合条件的项目，辅导员审核确认最终等级。
8. 公示查询：系统提供公开公示页，未登录用户也可查询已发布的获奖结果。
9. 数据导入导出：管理员可下载模板、批量导入学生名单与课程成绩，并导出学生账号。

### 3.3 非功能需求

1. 易部署：本地运行无需额外安装数据库，H2 文件数据库自动初始化。
2. 易维护：后端按 controller、service、mapper、entity 分层组织，前端按 layout 和 page 拆分。
3. 安全性：接口基于 JWT 与角色注解控制访问权限，密码使用 BCrypt 加密存储。
4. 可扩展性：评分规则、奖学金等级、项目条件和数据导入逻辑集中在服务层，便于后续扩展。
5. 用户体验：前端采用 Ant Design 组件库，针对不同角色提供独立导航与页面。

```text
浏览器
  |
  | React + Ant Design 前端页面
  |
  | RESTful API / JWT
  |
```

## 六、核心业务流程

### 6.1 登录认证流程

用户通过账号和密码登录。系统先查询 `users` 表判断账号是否存在和是否启用；若用户已修改密码，则使用 BCrypt 校验密码哈希；若尚未修改密码，则使用 `school_auth_mock` 中的初始密码进行校验。校验通过后，后端生成 JWT，前端保存 token 并在后续请求中自动携带。

### 6.2 综合测评流程

1. 学生进入综合测评页面，系统自动创建或读取当前学年综测记录。
2. 学生填报德育、素质拓展、创新模块材料。
3. 每新增或修改一条材料，系统调用评分服务计算单项分。
4. 系统重算德育、智育、素质拓展、创新四个模块分数。
5. 学生提交综测后，记录状态变为 `SUBMITTED`。
6. 辅导员审核材料，审核结果会影响最终综测分数。

### 6.3 奖学金评选流程

1. 管理员创建奖学金项目，配置一等、二等、三等奖学金比例和金额。
2. 学生完成综测填报，辅导员完成材料审核。
3. 管理员执行“排名与等级分配”。
4. 系统重算所有学生综测分，按总分降序排名；若总分相同，以创新分作为破平依据。
5. 系统根据各等级比例计算名额，并从高到低分配推荐等级。
6. 学生提交奖学金申请，系统写入综测分数、排名和推荐等级快照。
7. 辅导员审核申请并确认最终授予等级。
8. 管理员发布公示，公开页面展示获奖名单。

## 七、核心算法设计

### 7.1 综合测评总分模型

系统综合测评总分公式为：

```text
S = W1 * S1 + W2 * S2 + W3 * S3 + S4
```

其中：

| 模块 | 含义 | 学术型权重 | 专业型权重 |
|------|------|------------|------------|
| S1 | 德育分 | 25% | 25% |
| S2 | 智育分 | 45% | 40% |
| S3 | 素质拓展分 | 30% | 35% |
| S4 | 创新分 | 直接加分 | 直接加分 |

系统在 `EvaluationService` 中根据学生培养类型选择不同权重，并将总分保留三位小数。

### 7.2 德育模块计分

德育模块支持志愿服务、处分扣分、导师评议、研究生代表评议、辅导员评议等项目：

1. 志愿服务按服务时长折算，满 4 小时计 4 分，不足 4 小时按规则补充分值，最高 10 分。
2. 处分按不同等级映射为负分。
3. 评议类项目直接采用填报或评议分值。

### 7.3 智育模块计分

智育分根据课程成绩和学分计算加权平均：

```text
智育分 = sum(课程成绩 * 学分) / sum(学分)
```

该逻辑可避免单纯平均导致高低学分课程权重不一致的问题。

### 7.4 素质拓展模块计分

素质拓展模块覆盖学生干部任职、竞赛、荣誉、国际交流、创业等项目。系统根据项目类型、级别、奖项等级、任职时长和是否团队成果计算得分。例如：

1. 任职得分由岗位等级和类别共同决定，任职不足 6 个月不计分，6 至 12 个月按 50% 计分。
2. 竞赛按国家级、省级、校市级及一二三等奖映射分值。
3. 团队成果按 60% 系数折算。

### 7.5 创新模块计分

创新模块覆盖学科竞赛、专利、科研项目、论文、顶会等成果。系统根据基础分、竞赛类别系数和多人合作系数计算最终得分：

```text
创新单项分 = 基础分 * 类别系数 * 合作系数
```

其中，A/B/C 类竞赛系数分别为 1.2、1.0、0.5；多人合作成果通过 `CollabCoefficient` 按作者总数和本人排名计算贡献系数；若导师参与，系统会自动调整学生作者总数和排名。

### 7.6 排名与等级分配算法

奖学金等级分配由 `RankingService.rankAndAssign(projectId)` 实现，主要步骤如下：

1. 读取奖学金项目及对应学年。
2. 对所有学生执行综测重算，保证排名使用最新分数。
3. 查询该学年全部综测记录。
4. 按综合测评总分降序排序，总分相同时按创新分降序排序。
5. 写入学生总体排名。
6. 按每个奖学金等级配置的比例计算名额：

```text
名额 = floor(总人数 * 比例 / 100)
```

7. 按等级顺序自上而下分配推荐等级。
8. 更新已有申请记录的推荐等级、总分快照和排名快照。
9. 将项目状态更新为 `REVIEWING`，标记已完成排名。

该算法将排名、比例名额和等级推荐固化在系统中，减少人工切分名额带来的误差。

## 九、安全设计

### 9.1 身份认证

系统使用 JWT 作为登录凭证。用户登录成功后，后端签发包含用户 ID、账号、角色和姓名的 token。前端通过 Axios 请求拦截器携带 token，后端通过认证拦截器解析 token 并写入认证上下文。

### 9.2 权限控制

后端通过 `@RequireRole` 注解限制接口访问角色。例如，学生接口只允许 `STUDENT` 访问，管理员接口主要由 `ADMIN` 访问，部分审核和查询接口允许 `COUNSELOR` 或 `ADMIN` 访问。前端也通过路由守卫根据用户角色控制页面访问。

### 9.3 密码安全

用户首次登录可使用学校统一身份认证模拟表中的初始密码。修改密码后，系统使用 BCrypt 对新密码进行哈希存储，不保存明文密码。

### 9.4 数据隔离

学生端接口通过当前登录账号定位学生档案，并在修改、删除材料时校验记录归属，防止操作他人数据。辅导员审核场景按所带学生范围进行业务约束，管理员拥有全局管理权限。

## 十二、项目特色

1. 贴合真实业务：围绕研究生综合测评和奖学金评选的实际流程设计，覆盖填报、审核、排名、公示完整闭环。
2. 规则自动化：将复杂的综测权重、材料计分、创新成果系数和奖学金比例分配固化到服务层，降低人工计算错误。
3. 多角色协同：学生、辅导员、管理员分别拥有独立页面和业务权限，职责边界清晰。
4. 可快速演示：H2 数据库和初始化脚本降低部署门槛，适合课程答辩、项目展示和本地测试。
5. 易扩展维护：评分服务、排名服务、导入服务和控制器职责分离，后续可扩展更多奖学金类型、审批环节或统计报表。

## 十三、存在不足与改进方向

当前系统已实现奖学金评选核心流程，但仍有进一步完善空间：

1. 审计日志表已设计，但可进一步补充关键操作记录，如审核、排名、发布公示、数据导入等。
2. 可增加更细粒度的辅导员数据范围控制，例如按学院、专业、年级或班级绑定管理权限。
3. 可完善单元测试和集成测试，重点覆盖评分规则、排名算法、权限拦截和导入逻辑。
4. 可增加消息通知能力，在材料被驳回、申请审核通过或公示发布时提醒学生。
5. 可支持 MySQL、PostgreSQL 等生产数据库，并通过配置切换不同运行环境。
6. 可增加操作留痕和版本快照，便于复核历史评分规则和奖学金评选结果。

## 十四、总结

本项目完成了一个基于 Spring Boot 和 React 的研究生奖学金评选系统，实现了从综合测评填报、材料审核、奖学金项目配置、自动排名与等级分配到公示查询的完整业务闭环。系统通过规则化评分和自动化排名提高了评选效率与准确性，通过 JWT 认证和角色权限控制保证了基本安全性，通过 H2 数据库和演示数据降低了部署与演示成本。

整体来看，该系统具备较完整的业务覆盖度、清晰的工程分层和良好的扩展基础，能够满足研究生奖学金评选信息化管理的基本需求，并为后续接入真实统一认证、生产数据库、消息通知和更复杂的评审流程提供了基础。
